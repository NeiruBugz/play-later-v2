## Divergence Call-outs (waive rows → `CLAUDE.md` "Known gaps")

**#D1 — SidebarNavLinks: shadcn `Sidebar` primitive not used (row 3)**
Canonical uses the full shadcn `Sidebar` provider + `SidebarMenuButton` with collapsible-icon mode and tooltip-on-collapse. TanStack sidebar is a hand-rolled `<aside>` with TanStack `<Link>`. Rationale: the shadcn `Sidebar` is a heavyweight primitive (>500 lines) with cookie-backed collapse state, provider context, and Radix sub-primitives. The tanstack app does not need collapsible-icon mode at this stage — persistent full-width sidebar is correct for the current layout. Adopting the shadcn `Sidebar` would require provider wiring in `__root.tsx`, a layout restructure, and a new `Sidebar` primitive in `src/shared/ui/` with no other consumer. Intentionally waived; revisit if collapse mode becomes a requirement.

**#D2 — AddGameTrigger: quick-add Popover path absent (row 13)**
Canonical ships two add-game surfaces: a full `Dialog`-based form and a `Popover`-based "quick add" triggered from the library card hover state. TanStack ships only the `Dialog` path via `AddGameTrigger`. Rationale: the `Popover` quick-add is a progressive-enhancement convenience, not a primary flow. The tanstack library card does not yet have the hover-state CTA that would host a quick-add trigger. Port the Popover quick-add path when card hover actions are built.

**#D3 — LibraryModal: no desktop/mobile split, no cover thumbnail in header (rows 18, 19)**
Canonical ships separate `DesktopLayout` / `MobileLayout` components and renders the game's cover thumbnail + title in the modal header (`library-entry-metadata.tsx`). TanStack ships a single responsive `Dialog` form with `DialogTitle` text only. Rationale documented in CLAUDE.md Slice 11: collapsing to one form simplifies code without user-visible harm; the game title in `DialogTitle` satisfies the accessibility requirement; cover thumbnail adds visual richness but not functional parity. Both are intentional simplifications. Revisit if design review deems the cover thumbnail load-bearing for context.

## Recommended Sequencing Note

The 8 primitives divide into two dependency tiers. **Tier 1 (no inter-primitive dependencies, port in parallel):** `DropdownMenu`, `Select`, `Badge`, `Tooltip`. `DropdownMenu` unblocks four rows (4, 9, 20, 21) including the high-visibility `SidebarUserMenu` and `LibraryCardMenu`. `Select` unblocks the filter sidebar (row 6) and the library modal status/sort fields (rows 15 and implicitly sort in row 6). `Badge` unblocks status-count chips in the filter sidebar (row 6). `Tooltip` unblocks the related-game card (row 26 — low priority). **Tier 2 (start in parallel with Tier 1, but block the larger GREEN tasks):** `Popover` + `Command` (needed together for `PlatformFilterCombobox` in rows 6 and 16); `Sheet` + `RatingInput` (needed together for `MobileFilterBar` in row 7; `RatingInput` is also needed for rows 10, 17, 21). The GREEN filter sidebar rebuild (row 6 task) should start only after `Select`, `Badge`, `Popover`, and `Command` are all in `src/shared/ui/`. The GREEN mobile filter bar (row 7 task) should start only after `Sheet` and `RatingInput` are in `src/shared/ui/`. The GREEN sidebar user-menu (row 4 task) and card-menu (row 20 task) share `DropdownMenu` and can be worked on the same PR or back-to-back.
