SavePoint
// MOBILE UX AUDIT

Respecting the player on a small screen.

SavePoint is PWA-ready, which means the phone is no longer a courtesy view — it's a first-class home for the library. This audit walks every core surface as a patient gamer would hold it one-handed, names where the desktop layout was simply shrunk rather than rethought, and sets the principles for a mobile-first rebuild we can then scale back up to desktop.

Scope
8 core surfaces · phone widths (390–430px)
Source
savepoint-tanstack · FSD widgets
Severity
19 findings · 2 critical · 6 high
Prepared
June 2026
// METHOD & SEVERITY

How we read each screen

Findings come from the shipped widget code, not guesses — every issue cites the component it lives in. Each is rated by how much it blocks a patient gamer from doing the one thing the app is for: curating a collection and logging what a game meant.

Critical

A primary task is unreachable or broken on a phone. Fix before PWA launch.

High

The task works but fights the user — wasted space, lost actions, friction.

Medium

Noticeable rough edge that erodes the cozy, unhurried feel of the product.

Low

Polish — small tap targets, vestigial desktop affordances, visual nits.

// 01 · GLOBAL NAVIGATION

The tab bar shows three of five rooms

The desktop sidebar lists Library, Search, Journal, Profile, and Settings, plus a command palette and an add-game button. The mobile bottom nav (app-bottom-nav.tsx) carries only Library, Journal, and Profile — and the app's single most important verb, "log a session," has no home in the chrome at all.

NAV-01

Dashboard is the post-login home, but unreachable from the tab bar

Critical

Sign-in lands the user on /dashboard — the quick-log hero, "Playing," and "Up Next." Yet the bottom nav has no Home tab, so once a user taps Library or Journal there is no way back to the dashboard except the browser. In a standalone PWA there is no browser chrome to fall back on.

Where
widgets/app-bottom-nav — NAV_ITEMS = Library, Journal, Profile
Impact
The app's landing surface becomes a dead end after first navigation.
Fix
Add a Home tab as the first item. Reserve the tab bar for the five true destinations.
NAV-02

"Log a session" — the core verb — has no global entry point

High

The product's tagline is "Log tonight's session." But logging only exists inside a game-detail playthrough (LogSessionDrawer) and adding a game is a FAB that lives only on the Library page. From the dashboard, journal, or profile there is no one-tap way to capture a session.

Where
FAB scoped to widgets/library-page; log buried in widgets/game-detail
Impact
The reflective habit the app is built around takes 3–4 taps from most screens.
Fix
Promote a center "Log" action into the tab bar (or a persistent FAB) that opens the log sheet from anywhere.
NAV-03

Top-bar icon buttons are below the 44px touch minimum

Medium

Search and theme-toggle in the mobile top bar are rendered at h-9 w-9 (36px). Apple's HIG and WCAG target a 44px minimum; 36px is a frequent mis-tap on the move.

Where
widgets/app-mobile-topbar — h-9 w-9 hit areas
Fix
Bump tappable chrome to 44px; keep the 20px glyph centered inside.
NAV-04

Active tab signalled by color alone

Low

The active tab only swaps text color to text-foreground. There's no filled icon or indicator, so the "you are here" read is weak — and color-only states are an accessibility risk.

Fix
Fill the active icon and/or add a small accent dot or pill behind it.
NAV-05

Global search collapses to a single icon; no command palette equivalent

Low

Desktop's ⌘K command palette is the fast path to any game. On mobile it degrades to a search icon → full page. Acceptable, but the rich "jump to anything" affordance is lost rather than reimagined.

Fix
Let the Search tab open an inline search sheet with recents, not a cold page nav.
// 02 · DASHBOARD

A two-column desk, stacked into a long scroll

The dashboard's lg:grid-cols-[1fr_1fr] collapses cleanly to one column on a phone — but nothing re-prioritizes. Stats, "Playing," "Up Next," and "Recently added" all stack at equal weight, so the thing a returning player wants most (continue a session) competes with everything else.

DASH-01

No mobile hierarchy — every section weighs the same

High

On desktop the eye can scan two columns at once. Stacked on a phone, the quick-log hero, a stats card, and three game rails become a uniform vertical list. The primary "continue playing" intent is four scroll-lengths from secondary stats.

Where
widgets/dashboard-page — flat stack below the hero
Impact
Returning players scroll past their most-wanted action.
Fix
Lead with a compact "Jump back in" card; demote stats to a single inline strip; make rails horizontally scrollable so each costs one screen height, not three.
DASH-02

Game rails wrap into tall grids instead of swiping

Medium

Each DashboardGameSection renders a grid that, narrowed to one or two columns, becomes a tall block. Four sections of tall blocks is a very long dashboard.

Fix
Convert rails to horizontal snap-scroll carousels — a native, space-efficient mobile pattern that also signals "there's more."
// 03 · LIBRARY

One game per row wastes the whole screen

The library grid is grid-cols-1 below the md breakpoint — a single fat card per row on a phone. For an app whose whole pitch is "a library, not a backlog," browsing the collection is the marquee activity, and one-up makes a 30-game shelf feel endless. The filter sheet itself is solid; the density and a few vestigial desktop affordances are the misses.

LIB-01

Single-column grid on phones

High

Cover art is the library's visual language. At one column, a phone shows ~2 games per screen; at two columns it shows 6+, matching how every console and storefront presents a library.

Where
widgets/library-page — grid-cols-1 md:grid-cols-[…]
Impact
Browsing feels like an infinite scroll instead of a glanceable shelf.
Fix
Default to a 2-up cover grid on phones; offer a list-density toggle for those who want metadata-forward rows.
LIB-02

Status filter lives behind a sheet, with no at-a-glance switch

Medium

Status (Playing / Up Next / Shelf / Played / Wishlist) is the primary lens on a library, yet on mobile it's one of many controls inside the filter Sheet. Switching "what am I playing?" → "what's up next?" is a multi-tap round trip.

Where
features/filter-library — MobileFilterBar Sheet
Fix
Surface status as a sticky, horizontally-scrollable segmented row above the grid; keep secondary filters (platform, rating, sort) in the sheet.
LIB-03

Filter breakpoint (xl) disagrees with chrome breakpoint (md)

Medium

The bottom nav / top bar switch at md (768px), but the filter sidebar↔sheet switch at xl (1280px). Between those widths a user gets desktop chrome but the mobile filter sheet — an inconsistent, untested middle state.

Fix
Pick one responsive breakpoint system and apply it across chrome and filters.
LIB-04

Keyboard-only affordances ship to touch devices

Low

The title-filter input shows a / keyboard-shortcut hint and the "Imported games" link is a small inline text target — both are desktop habits that add clutter or mis-taps on a phone.

Fix
Hide the / kbd hint on touch; give "Imported games" a full-height tappable row in an overflow menu.
// 04 · GAME DETAIL

A bento grid becomes an endless ribbon

Game detail is the densest screen in the app: a screenshot hero, status switcher, playthroughs, about, themes & tags, journal, related games, screenshots, times-to-beat. On desktop these tile into a 1.35fr / 1fr bento. On a phone every card collapses into a single tall column with no way to jump between sections — and the primary action scrolls away.

GD-01

The primary action scrolls off and never comes back

Critical

The LibraryStatusSwitcher and log/track CTAs live inline in the hero. Once a user scrolls past the fold to read the about panel or journal, the one action they came to take — change status, log a session — is gone, with no sticky bar to recall it.

Where
widgets/game-detail — actions inside #hero section
Impact
On the longest scroll in the app, the main verb is only reachable at the very top.
Fix
Pin a slim action bar to the bottom (above the tab bar) with the status pill + "Log session," persistent through scroll.
GD-02

Seven+ stacked panels with no in-page navigation

High

Collapsed to one column, the bento becomes a 7–9 card ribbon. There's no section index, no tabs, no jump — finding "Related games" means scrolling past everything else every time.

Where
game-detail-bento-grid — md:grid-cols-[1.35fr_1fr] → single column
Fix
Group panels under a sticky segmented control (Overview · Playthroughs · Journal · About) or collapsible accordions so the page has a navigable spine.
GD-03

Hero crowds the title against the critic-score ring

Medium

The hero lays the text-h1 title and CriticScoreRing in a justify-between row with a fixed pt-[140px]. On a 390px screen a long title wraps awkwardly beside the ring, and the fixed top padding is tuned for desktop backdrop height.

Fix
Stack hero elements vertically on mobile: cover, then title, then a metadata row that carries the score as a chip.
GD-04

Breadcrumb consumes a row of little value

Low

"Library / Games / Title" is a desktop wayfinding convention; on mobile the back gesture and tab bar already cover it, so it mostly steals vertical space at the top of the densest page.

Fix
Replace with a single back affordance in the top bar.
// 05 · JOURNAL

Reflection deserves a sheet, not a centered modal

The timeline list reads well on mobile. The friction is in composing: "reflect, don't review" is the heart of the app, but the compose flow uses a centered Dialog — a desktop modal — which on a phone fights the keyboard and crops the writing area.

JRN-01

Compose uses a centered Dialog, not a full-height sheet

High

ComposeJournalEntryDialog is a centered modal. When the keyboard opens on a phone it covers most of the dialog, leaving a small textarea boxed in the middle — the opposite of an inviting space to reflect. (Log session already uses a Drawer; compose should match.)

Where
features/compose-journal-entry — Dialog
Fix
Use a full-height bottom sheet that grows with the keyboard, with the text area as the hero and "playtime is enough, thoughts optional" guidance inline.
JRN-02

Entries don't group by game on mobile

Medium

The timeline is a flat chronological list (a documented divergence). On a small screen, a per-game grouping with a cover thumbnail would give the scroll rhythm and let a player relive one world at a time.

Fix
Optionally cluster consecutive same-game entries under a slim cover header.
// 06 · PROFILE & SETTINGS

Desktop rails that should become native lists

Profile and Settings both lean on side-by-side layouts — a settings rail beside content, a tabbed profile beside stats. They collapse without breaking, but they don't feel like a phone.

SET-01

Settings rail stacks above content instead of becoming a list

Medium

The settings-rail is a vertical category nav meant to sit beside a content pane. Stacked on mobile, you scroll past the whole nav to reach the form. Phones expect a grouped list → drill-in detail pattern.

Where
widgets/settings-rail
Fix
Adopt an iOS-style grouped settings list (Profile, Account, Appearance, Imports) that pushes into each detail screen.
PRO-01

Profile tabs + stats compete for the fold

Medium

Banner, avatar, follow/edit action, stat counts, and Overview/Library/Activity tabs all sit in the header region. On a phone this pushes the actual content (the library or activity) well below the fold.

Where
widgets/profile-overview
Fix
Compress the header to avatar + name + one stat row + a sticky tab strip; let content start higher.
PRO-02

Header actions may overflow a narrow row

Low

Edit-profile / follow buttons injected via headerActions share a row with the name and stats; on the narrowest screens this risks truncation or wrapping.

Fix
Give the primary action a full-width button below the identity block on mobile.
// THE REBUILD

Five principles for the mobile-first pass

Every redesigned screen in the companion file follows these. They're chosen so that scaling back up to desktop is additive — more columns, more room — never a different product.

01

The thumb owns the bottom third

Primary navigation and the core "Log" verb live in the bottom chrome, reachable one-handed. Nothing critical hides at the top of a long scroll.

02

Cover art is the interface

The library reads as a shelf, not a feed — at least two covers wide — because browsing the collection is the point of the product.

03

Dense screens get a spine

Game detail and profile use sticky segmented controls or carousels so a long page is always navigable, never an undifferentiated ribbon.

04

Sheets, not modals

Logging and reflecting happen in bottom sheets that rise with the keyboard — calm, full-width, and native to the platform.

05

Cozy is a constraint, not decoration

Warm cream surfaces, generous line-height, sentence-case copy, 44px targets. The mobile build should feel as unhurried as the desktop one.
